### Detailed Implementation:

#### 1. Review the internship as one connected architecture journey

Rather than treating each week as isolated content, I reviewed the full internship as one continuous technical progression:

* secure account setup
* networking design
* compute deployment
* storage usage
* IAM and observability
* private and hybrid S3 connectivity
* policy-based access control
* documentation and reporting

This made the final report feel more complete and realistic as an engineering learning journey.

#### 2. Validate technical consistency across all sections

I checked whether service relationships described in the report were consistent throughout, for example:

* EC2 placement inside the VPC
* use of IAM Role instead of static credentials
* S3 access through Gateway or Interface Endpoints
* Route 53 participation in private DNS resolution
* endpoint policy restrictions on S3 access

This review was necessary because technical reports can become inconsistent if each section is written separately without a final pass.

#### 3. Clean up resources as part of responsible cloud practice

I revisited resources that may continue costing money or create unnecessary clutter, such as:

* EC2 instances
* NAT Gateway
* Interface Endpoints
* S3 resources used only for tests

This turned cleanup into part of the learning outcome, not just an afterthought.

#### 4. Reflect on future learning direction

After reviewing the full 12-week experience, I identified the technical areas I want to deepen further:

* cloud networking
* cloud security
* hybrid connectivity
* automation and Infrastructure as Code

This reflection helped connect the internship outcome to my next learning goals rather than ending only with a completed report.
